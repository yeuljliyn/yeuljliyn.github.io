About Me

새로운 기술을 배우는 일보다, 그 기술로 원하는 결과를 만드는 일이 더 재미있는 개발자입니다.

제가 개발에 관심을 갖게 된 계기는 마인크래프트라는 게임이었습니다. 게임을 플레이하는 것보다 레드스톤 회로를 만들고 명령어를 조합해서 새로운 기능을 구현하는 시간이 훨씬 즐거웠습니다. 버튼 하나를 눌렀을 뿐인데 제가 만든 장치가 의도한 대로 움직이고, 생각했던 시스템이 실제로 동작하는 모습이 신기했습니다. 그때 처음으로 '직접 만드는 일'이 정말 재미있다는 걸 느꼈고, 자연스럽게 개발을 시작하게 되었습니다.

지금도 개발하는 방식은 크게 달라지지 않았습니다. 처음 보는 기술이 나오면 겁부터 먹기보다 일단 만들어 봅니다. AI와 공식 문서를 참고해서 간단한 프로토타입을 만들고, 실행해 보면서 하나씩 고칩니다. 오류가 생기면 원인을 찾고, 해결하면 또 다음 문제를 만나는 과정이 저한테는 스트레스보다 재미에 가깝습니다. 그래서 새로운 기술을 배울 때도 공부만 오래 하기보다 직접 부딪히면서 익히는 편입니다.

개발하면서 가장 기분 좋은 순간은 제가 만든 결과물을 다른 사람이 사용했을 때입니다. 프로젝트를 여러 번 수정하면서 완성도를 높이고, "재밌다", "생각보다 퀄리티가 좋다"는 피드백을 받을 때마다 다시 개발할 힘을 얻습니다. 그래서 저는 코드를 많이 작성한 날보다, 사용자가 만족하는 결과물을 만든 날을 더 기억합니다.

Problem Solving

문제가 생기면 답을 찾기보다, 직접 해결 방법을 만들어 보는 편입니다.

NC AI VARCO Voice 공모전에서 Unity 플러그인을 만들면서 가장 크게 느꼈습니다. 제공된 API 문서는 대부분 Python 예제로 작성되어 있었지만 프로젝트는 Unity(C#) 환경에서 개발하고 있었습니다. 단순히 문법만 바꾸면 될 줄 알았지만 실제로는 요청 방식, 데이터 처리, 비동기 흐름까지 전부 다시 맞춰야 했습니다.

처음에는 AI가 알려주는 방법대로 구현했지만 제대로 동작하지 않았습니다. 그래서 접근 방식을 바꿨습니다. Python 코드가 어떤 순서로 동작하는지 먼저 분석한 뒤 요청, 응답, 음성 생성 과정을 각각 분리해서 하나씩 구현했습니다. 작은 기능이 정상적으로 동작하면 다음 단계로 넘어가는 식으로 계속 검증했습니다.

몇 번이나 처음부터 다시 만들었지만 결국 Unity 안에서 바로 사용할 수 있는 플러그인을 완성했습니다. 이 경험 이후로는 처음 보는 기술이 나와도 "못 한다"는 생각보다 "어떻게 쪼개면 만들 수 있을까?"부터 생각하게 되었습니다.

Communication

좋은 협업은 말을 많이 하는 것이 아니라, 같은 그림을 보는 것이라고 생각합니다.

프로젝트를 하다 보면 기획서만 보고 개발하기 어려운 경우가 많았습니다. 같은 문장을 읽어도 사람마다 생각하는 결과가 달랐고, 실제로 구현이 끝난 뒤 기획자의 의도와 전혀 다른 기능이 나온 적도 있었습니다.

그 일을 겪고 나서는 개발을 시작하기 전에 제가 이해한 내용을 다시 설명하는 습관을 만들었습니다. "이 기능이 이런 방식으로 동작하는 것이 맞나요?"를 먼저 확인했고, 애매한 부분은 간단한 프로토타입을 만들어 기획자와 함께 보면서 방향을 맞췄습니다.

기획자도 이전보다 예외 상황과 세부 동작을 자세하게 작성하기 시작했고, 큰 수정이 필요한 경우가 눈에 띄게 줄었습니다. 그 경험 이후 저는 좋은 소통은 설명을 잘하는 것보다 서로 같은 결과를 떠올리고 있는지 확인하는 과정이라고 생각하게 되었습니다.

Resource Management

반복되는 작업을 발견하면, 사람보다 도구가 하는 것이 맞는지 먼저 생각합니다.

리듬게임 프로젝트에서는 스테이지마다 노트를 하나씩 입력해야 했습니다. 개발자가 계속 데이터를 입력하는 방식은 시간이 오래 걸렸고, 기획자도 원하는 패턴을 직접 수정하기 어려웠습니다.

그래서 Unity 에디터 안에서 사용할 수 있는 비트메이커 툴을 만들었습니다. 기획자가 직접 노트를 배치하고 바로 플레이해 볼 수 있는 도구였습니다.

도구를 만든 이후에는 기획자가 직접 데이터를 만들 수 있게 되었고, 저는 반복 작업 대신 게임 시스템 개발에 집중할 수 있었습니다. 한 번 만든 도구는 이후 스테이지에서도 그대로 사용할 수 있었기 때문에 프로젝트가 진행될수록 작업 속도도 자연스럽게 빨라졌습니다.

예전에는 개발 속도를 높이는 방법만 고민했다면, 지금은 반복되는 작업 자체를 없애는 방법을 먼저 고민합니다.

User Experience

기능이 완성됐다고 생각하는 순간은 코드가 끝났을 때가 아니라, 좋은 반응을 들었을 때입니다.

NC AI 공모전 프로젝트에서는 컷신 연출과 구현을 맡았습니다. 이전부터 꼭 해보고 싶었던 분야였지만 제대로 만들어 본 것은 처음이었습니다. 카메라 움직임, 캐릭터 동선, 음성 타이밍까지 계속 수정하면서 조금이라도 더 자연스럽게 보이도록 여러 번 손봤습니다.

최종 발표가 끝난 뒤 "학생들이 만든 프로젝트라고 생각하지 못할 정도로 퀄리티가 좋았다."는 평가를 들었습니다. 특히 컷신 이야기가 나왔을 때 가장 기뻤습니다. 제가 직접 연출한 장면이 다른 사람에게도 그대로 전달됐다는 뜻이었기 때문입니다.

그 이후부터는 새로운 기능을 만드는 것보다 사용자가 어떻게 느끼는지를 더 많이 생각하게 되었습니다. 결국 개발은 코드를 보여주는 일이 아니라 경험을 만드는 일이라고 생각합니다.

Growth

한 가지 문제에 끝까지 몰입하는 것이 제 장점이자 가장 큰 단점이었습니다.

문제가 생기면 해결할 때까지 계속 붙잡고 있는 편입니다. 덕분에 어려운 오류도 많이 해결했지만, 반대로 하나의 문제에 너무 오래 시간을 쓰는 경우도 많았습니다.

이 습관을 바꾸기 위해 작업을 최대한 작은 단위로 나누기 시작했습니다. 일정 시간을 정해 놓고 작업한 뒤에는 현재 상태를 기록하고, 다음 작업으로 넘어갈지 계속 진행할지 다시 판단했습니다.

처음에는 흐름이 끊기는 것 같았지만 오히려 프로젝트 전체를 보는 시야가 생겼습니다. 지금은 문제 하나를 빨리 해결하는 것보다, 프로젝트 전체를 끝까지 완성하는 것이 더 중요하다고 생각합니다.
